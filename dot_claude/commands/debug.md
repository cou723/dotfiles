```手順コード

ユーザーにあなたが考える「「$ARGUMENTS」を完了したと思える条件」を提示
「「$ARGUMENTS」を完了したと思える条件」の検証方法を提示

「$ARGUMENTS」の解決の難易度を低・中・高に分類
  低：これはユーザーの要求がそのままエラーメッセージとして出ていたり、同じプロジェクト内に似たようなコードがある場合に適応されます
  中：これはユーザーの要求に基づいていくつかのエラーメッセージやログを調査する必要があったり、プロジェクトの初期段階による開発環境がおかしい場合（類似のコードがほぼない場合）が含まれます。基本的にプロジェクトの最初のほうのデバッグ要求は大体ここに属します。
  高：これはユーザーの要求だけでは状況があいまいで、かつ、現在のエラーの状態、ログの状態とユーザーの要求の一貫性が見えない場合に属します。まったく修正方法の検討がつかず、地道なデバッグ作業が必要な場合に含まれます。インターネットで同じ案件があった場合難易度は中が適正ですが、基本的には微妙に違うため、この場合は地道なデバッグ作業が必要とされます。

if (難易度が低である) {
    修正
    while (明示した検証方法を用いて完了したといえない) {
        コードを修正
    }
    ユーザーに完了内容を提示
    終了
}
else if (難易度が中、高である) {
    while (バグの原因が明確にわかり、絶対にこれだという自信が持てない) {
        バグの仮説とその検証方法を1~3個形成、提示する
            この際ある程度インターネットの記事を参考にする
        while (許可が取れなかった) {
            仮説と検証方法の修正
        }
        if (gitリポジトリである) {
            チェックポイントとして仮のコミットを作成
        }
        仮説をもとに検証する
            この際によくわからなくなった場合は先ほどのコミットに戻る
    }
    バグの修正を行う    
    while (明示した検証方法を用いて完了したといえない) {
        コードを修正
    }
    ユーザーに完了内容を提示
    終了
}
```
